2. PHẠM VI KIỂM THỬ

2.1. In-scope (4–5 chức năng chính)

Theo yêu cầu, bộ kiểm thử tập trung vào 5 chức năng quan trọng để đạt hiệu quả và dễ báo cáo. Mỗi tình huống kiểm thử sẽ có đủ Black Box, Grey Box và White Box phù hợp.

No Chức năng Mô tả Priority
1 Authentication Register / Login / Logout; JWT token; password hashing; session persistence High
2 Menu Browse list; filter category; search; view details High
3 Cart Add/update/remove; empty cart; total calculation; sync with DB High
4 Order & Payment Create order; validate address; payment methods (COD/VNPAY/Stripe/MoMo); order history & tracking High
5 Admin Food management; order management; update status Medium

2.2. Out-of-scope

  • Mobile app (nếu có)
  • Hạ tầng payment gateway nội bộ (chỉ test tích hợp)
  • Các tính năng chưa triển khai/không ổn định (nếu có)

3. CHIẾN LƯỢC KIỂM THỬ

3.1. Các loại kiểm thử

  • Black Box Testing: kiểm thử chức năng từ góc độ người dùng (theo TEST_CASES.md)
  • Grey Box Testing: kiểm thử dựa trên hiểu biết một phần về API/DB (JWT, hashing, constraints, API codes)
  • White Box Testing: kiểm thử logic code, coverage (statement/branch/path), security checks

3.2. Phương pháp luận (rút ra)

Phương phápMô tảÁp dụng
Equivalence Partitioning Chia input thành các lớp tương đương (valid/invalid) để giảm số test nhưng vẫn bao phủ tốt. Auth, Menu Search, Profile, Payment fields
Boundary Value Analysis Test tại biên (min, max, min-1, max+1): quantity, password length, name length… Cart quantity, Password length, Food price/name
Statement Coverage Đảm bảo các câu lệnh quan trọng được execute (mục tiêu ≥ 80%). Auth, Cart, Order services/controllers
Branch Coverage Đảm bảo các nhánh điều kiện (if/else) được test (mục tiêu ≥ 75%). Auth, Order creation, Payment verify
Path Coverage Test các đường đi chính trong flow (item mới vs đã có, payment success vs fail…). Cart add/update, Payment flows
Integration Testing Kiểm thử liên kết Frontend ↔ Backend ↔ DB và payment gateway callbacks. End-to-end order flow
Security Testing Test NoSQL injection, auth bypass, token expiry, validation đầu vào. Auth, Admin, Payment verify endpoints
Nguyên tắc thiết kế: Chỉ chọn 4–5 chức năng quan trọng và tập trung test theo tình huống (scenario-based), nhằm tối ưu thời gian nhưng vẫn chứng minh được phương pháp luận và độ bao phủ.

3.3. Tiêu chí hoàn thành & chấp nhận

  • Hoàn thành thiết kế test case cho 5 chức năng chính
  • Ưu tiên High phải có đủ test (positive/negative/boundary)
  • Không còn lỗi Critical; lỗi High được đánh giá và có kế hoạch xử lý
  • Coverage mục tiêu (nếu chạy tool): Statement ≥ 80%, Branch ≥ 75%
